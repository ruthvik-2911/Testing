## API Endpoints & Auth Flow

### 1. How a New User Gets a `uid`
The `uid` is not provided by the user, nor is it strictly tied to an external service like Firebase. It is **auto-generated by the backend's database schema**.

In the User model definition (`models/users.model.js`):
```javascript
const uuid_1 = require("uuid");

exports.UsersModel = new mongoose_1.Schema({
    uid: { type: String, default: () => (0, uuid_1.v4)() },
    // ... other fields
})
```
**What happens behind the scenes:**
1. When a user registers or gets created (e.g., via the `/user/create/:userType` endpoint, or when a new SUBSCRIBER logs in via phone for the first time), the backend calls `userModel.create(data)`.
2. The code explicitly does **not** pass a `uid`. 
3. Because `uid` is missing, Mongoose triggers its `default` function which calls `uuid.v4()`, generating a universally unique 36-character string.
4. This auto-generated `uid`, along with the MongoDB internal `_id`, is permanently saved to the user's database record.

### 2. How the JWT Token is Generated
Once the user exists in the database and they attempt to log in (either via email/password or by verifying an OTP):

1. **Authentication:** The backend verifies the credentials (e.g., checks if the password hash matches, or if the OTP is correct).
2. **User Retrieval:** Upon successful authentication, the backend fetches the complete user record from the database.
3. **Token Creation:** The backend calls `this.helper.generateJWT({ uid: user.uid, _id: user._id })` in `user.provider.js`.
4. **Signing:** Inside the `HelperService`, the `@nestjs/jwt` library takes that payload and signs it using the secret key (`JWT_SECRET` from `.env`).
5. **Response:** The resulting encrypted string (the JWT) is returned to the client in the API response.

### 3. Create New Company Endpoint

**Endpoint:**
```http
POST http://localhost:3000/v1/company
```

**Headers:**
```http
Content-Type: application/json
Authorization: Bearer <YOUR_JWT_TOKEN>
```

**Example JSON Payload:**
```json
{
  "name": "Acme Corp Services",
  "email": "contact@acmecorp.com",
  "companyType": "PRODUCTS_SERVICES", 
  "phoneNumber": {
    "countryCode": "+91",
    "dialNumber": "9876543210"
  },
  "companyLogo": "651a2b3c4d5e6f7a8b9c0d1e", 
  "companyCategories": [
    "651a2b3c4d5e6f7a8b9c0d1f"
  ], 
  "tax": {
    "taxType": "GST",
    "taxNumber": "22AAAAA0000A1Z5"
  },
  "billingAddress": {
    "addressLine1": "123 Business Avenue",
    "addressLine2": "Suite 400",
    "city": "Bengaluru",
    "state": "Karnataka",
    "zipCode": "560001",
    "country": "India"
  },
  "primaryContact": {
    "name": "John Doe",
    "email": "johndoe@acmecorp.com",
    "isSameAsBilling": true,
    "phoneNumber": {
      "countryCode": "+91",
      "dialNumber": "9876543210"
    },
    "alternativePhone": {
      "countryCode": "+91",
      "dialNumber": "9876543211"
    }
  }
}
```
**Notes:** 
- If `isSameAsBilling` is set to `false`, you must include an `"address"` object inside `primaryContact` with the same structure as `billingAddress`.
- Ensure `companyLogo` and `companyCategories` contain valid MongoDB ObjectIds if you're passing them.
